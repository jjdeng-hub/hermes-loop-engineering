# 内容创作与多平台分发工作流

## ⛔ 三条红线（违反任何一条直接拒绝或纠正）

1. **禁止虚构数据** — 任何图表、统计数据、对比数字必须有真实来源（GitHub API、平台公开数据等）。禁止凭感觉画柱状图/热度对比图。无法获取时明确说"这个数据我拿不到，建议你亲自验证"。
2. **禁止虚假对比** — 用户只深度用过工具 A 时，不要强行做"A vs B"对比。真诚告知局限性，聚焦 A 的优点展示。不打不公正的比较。
3. **禁止强迫视频** — 用户表达视频困难（配音/拍摄/剪辑）时，主动建议转为图文方案。图文生产效率更高，不要卡在半路上的视频。

## 核心原则

**一鱼多吃**：1 篇深度内容 → 5 个平台分发

**双向管道**：既有「原创内容 → 多平台」，也有「外文源 → 中文改编」。两条线独立运作，但共享分发环节。

**校准循环**：配合 `cheat-on-content` 技能，建立 打分→盲预测→发布→复盘→进化 的内容创作反馈闭环。避免"自娱自乐"式创作。具体流程：`初始化`（建立 state + rubric）→ `对标分析`（导入 3+ 对标账号数据）→ `打分`（7 维 rubric 评分 + composite）→ `盲预测`（cold-start 简化版，前 5 篇不给假精度数字）→ `发布` → `T+3 复盘`。详细对标分析模板见 `references/xiaohongshu-benchmark-analysis.md`。

```
网站教程（2000-3000 字，含代码/数据）
    │
    ├── 公众号文章（1500 字，精简版）
    ├── 小红书笔记 1（痛点+解决方案，3-5 张图）
    ├── 小红书笔记 2（实操步骤，3-5 张图）
    └── 小红书笔记 3（避坑指南，3-5 张图）
```

## 步骤

### 第 1 步：内容盘点

分析现有内容资产，确定：
- 已有教程/Skill 的数量和质量
- 内容缺口（哪些主题需要补充）
- 变现价值排序（引流向 / 核心向 / 转化向）

### 第 2 步：规划内容矩阵

按变现目标分类：

| 系列 | 定位 | 目标 | 频率 |
|------|------|------|------|
| 引流向 | 低门槛、场景化 | 小红书/公众号引流 | 每周 2-3 篇 |
| 核心向 | 深度实战、展示能力 | 筛选付费用户 | 每周 1 篇 |
| 转化向 | 变现路径、定价策略 | 引导购买服务/课程 | 每周 1 篇 |

### 第 3 步：写作网站教程

**结构模板**：

```markdown
# 标题：[痛点] + [解决方案] + [结果]

## 引言
- 痛点描述（具体场景）
- 解决方案概述
- 预期效果（量化）

## 核心内容（3-5 个案例/步骤）
### 案例/步骤 1
- 痛点
- 解决方案
- 工具推荐
- 代码/操作示例
- 效果（量化）

## 工具推荐清单
| 工具 | 用途 | 费用 | 难度 |

## 避坑指南
### ❌ 常见错误
### ✅ 正确做法

## 如何开始？
第 1 步...
第 2 步...

## 资源下载
- 完整代码包
- 配置指南
- 检查清单

## 相关教程
- 链接到已有内容
```

### 第 4 步：适配公众号（原创内容）

**精简要点**：
- 字数控制在 1500-2000 字
- 删除代码细节，保留核心思路
- 添加"点击查看完整教程"引导
- 格式：短段落 + 加粗重点

### 第 5 步（特殊流程）：英文商业内容 → 中文改编

适用于从英文技术博客、商业分析、行业报告等外文源改编为中文平台内容。**这个流程和「原创内容 → 多平台」是反向的**，有独立的规则。

#### 何时触发

- 用户发来一篇英文文章/PDF，要求"帮我看一下"
- 用户说"帮我改编成一篇公众号文章"
- 用户说"翻译一下这篇，改成适合国内读者的"

#### 核心工作流

**A. 深度阅读（先理解，不急着翻）**

通读全文，识别其骨架：
- **核心论点**：这篇文章在说什么？（一句话概括）
- **结构框架**：分几个阶段/章节？
- **数据支撑**：有哪些具体的数字、案例、时间线？
- **金句/亮点**：哪些段落值得保留原意？

**B. 决定改编策略**

| 原文特征 | 改编策略 |
|----------|----------|
| 结构清晰的指南类（如"90 天路线图"） | **保留框架**，替换内容 |
| 观点类/分析类 | **提取核心观点**，用中文表达 |
| 案例研究 | **保留结构**，但可补充中文案例 |
| 工具教程 | **替换工具**，用中文生态替代品 |

**C. 本地化变换矩阵**

| 维度 | 原文 | 中文版本 |
|------|------|----------|
| **定价** | $3,000-$15,000 | ¥5,000-¥50,000（按购买力平价 ×6-8 再÷2 调整） |
| **工具** | Claude Code, MCP, Cal.com, Loom, Stripe | Cursor/Claude, 飞书/微信, 腾讯会议, 微信视频, 微信支付 |
| **获客渠道** | LinkedIn, Twitter, Cold Email | 小红书, 朋友圈, 微信私域 |
| **平台** | Slack, Google Drive | 飞书/钉钉/企业微信 |
| **AI 术语** | agent, workflow, MCP server | 智能体, 工作流, 工具集成 |
| **文化参照** | "18 个月后竞争会变拥挤" | 保留，普适性洞察 |
| **文档语气** | 直接、自信、"这个窗口正在关闭" | 保留，但避免过于激进 |

**D. 价格本地化公式**

```
中国市场定价 ≈ 美国定价 × 6（汇率）× 0.3-0.5（购买力调整系数）

例：
$3,000 × 6 × 0.3 = ¥5,400   → 取整 ¥5,000
$15,000 × 6 × 0.5 = ¥45,000  → 取整 ¥50,000
$500 × 6 × 0.5 = ¥1,500     → 取整 ¥3,000/月（维护费）
```

**E. 工具替换清单**

| 英文生态 | 中文替代 | 说明 |
|----------|----------|------|
| Claude Code / Cursor | Cursor / Claude / 通义灵码 | AI 编程工具，Cursor 国内可用 |
| Notion | 飞书文档 / Notion | 飞书在国内更普及 |
| Cal.com | 腾讯会议 / 飞书日历 | 预约会议 |
| Loom | 微信视频 / 腾讯会议录制 | 异步视频沟通 |
| Stripe | 微信支付 / 支付宝 | 收款 |
| Slack | 企业微信 / 飞书 / 钉钉 | 团队沟通 |
| Google Drive | 百度网盘 / 阿里云盘 / 飞书云盘 | 文件管理 |
| LinkedIn | 脉脉 / 小红书 | 职业社交/内容分发 |
| Twitter / X | 微博 / 小红书 | 内容分发 |
| Medium | 公众号 / 知乎 | 深度内容平台 |
| Calendly | 腾讯会议预约 | 自动预约 |

**F. 公众号写作风格**

区别于小红书（短平快）和网站教程（结构化），公众号文章需要：

- **标题**：吸引点击但不过度夸张。好的中文公号标题通常有"悬念"或"反直觉"
- **开头**：直接抛出问题，用"每一天..."或"你有没有..."开头
- **段落**：短段落（每段 2-4 句），多空行，易读
- **加粗**：关键数据和结论加粗，方便扫读
- **引导关注**：文末可加引导语，但不强制
- **长度**：1500-2500 字，太长会掉阅读完成率
- **发布时间**：晚上 8-10 点最佳，工作日比周末好

**G. 改编示例**

```
原文：定价 $3,000-$5,000 per project
改为：一次性收取 5,000 到 20,000 元搭建费

原文：Claude Code or Cowork
改为：Claude / Cursor / 任何 AI 编程工具

原文：Cal.com for scheduling
改为：腾讯会议 / 飞书

原文：价格长期趋势
改为：保留，这是普适性商业洞察，国内同样适用
```

**H. 检查清单**

发布前检查：
- [ ] 价格已本地化（$ → ¥）
- [ ] 工具已替换（英文 SaaS → 中文生态替代品）
- [ ] 案例已检查（是否需要补充中文案例）
- [ ] 语气已调整（英文的直接 vs 中文的委婉）
- [ ] 字数符合平台要求（公众号 1500-2500 字）
- [ ] 长篇英文已经分段、加粗、加空行
- [ ] 文化梗/俚语已替换或删除

### 第 5 步：视频脚本创作（多平台适配）

适用于生产讲解/演示类视频内容。核心策略是**一个母版脚本，多平台裁剪输出**。

#### 触发条件

- 用户说"我想做一期视频分享[某个工具/概念]"
- 用户说"我想发多个平台"
- 内容适合录屏解说 + 旁白的形式

#### 核心原则

```
母版脚本（5-8 分钟）
    │
    ├── B站版（5-8 分钟，保留所有章节）
    ├── 小红书版（2-3 分钟，保留核心功能演示）
    └── 抖音/视频号版（30-60 秒，极速快节奏）
```

#### 结构模板

每个章节包含三个要素：**主讲稿 + 分镜 + 时长**。

```markdown
## 章节名（开始时间 - 结束时间）

🔥 所有平台必留 | 📦 B站完整版 | ✂️ 可裁剪

### 主讲稿
（逐字稿，照着念）

### 分镜
| 编号 | 画面 | 时长 |
|------|------|------|
| 1 | 录屏/镜头：具体操作 | Xs |

### BGM 建议（可选）
```

#### 章节设计原则

| 章节位置 | 时长占比 | 建议 |
|----------|----------|------|
| 开场 | 5-10% | 直接抛出问题/痛点，吸引注意力 |
| 功能/内容讲解 | 60-70% | 核心价值，逐个功能演示 |
| 亮点补充 | 15-20% | 差异化功能、彩蛋 |
| 安装/总结 | 5-10% | 引导关注/行动 |

#### 可裁剪标记系统

| 标记 | 含义 | 保留平台 |
|------|------|----------|
| 🔥 **必留** | 核心亮点，所有平台都保留 | B站 + 小红书 + 抖音 |
| 📦 **完整版** | 深度内容，只在长视频保留 | B站 |
| ✂️ **可裁剪** | 有价值但非核心，短视频可剪 | 小红书可酌情保留 |

#### 平台适配参数

| 维度 | B站 | 小红书 | 抖音/视频号 |
|------|-----|--------|------------|
| 时长 | 5-8 分钟 | 2-3 分钟 | 30-60 秒 |
| 比例 | 16:9 横屏 | 3:4 竖屏 | 9:16 竖屏 |
| 节奏 | 每个演示 ≤30s | 每个亮点 ≤15s | 每个镜头 3-5s |
| 字幕 | 建议加 | 必加 | 必加（大字） |
| 标题风格 | 技术向 | 干货向 | 快节奏/反差 |

#### 封面设计描述模板

```markdown
B站封面：
背景：[场景截图/纯色]
中央大字：[核心标题]
副标题：[一句话吸引]
右下角：[版本号/品牌]

小红书封面：
背景：[深/浅色渐变]
中央大字：[核心卖点]
副标题：[结果承诺]
```

#### 拍摄前检查清单

- [ ] 录屏工具已准备（建议 OBS Studio）
- [ ] 麦克风已测试
- [ ] 操作路径已确认（不允许录一半卡住）
- [ ] 窗口干净（关闭无关通知/标签页）
- [ ] 演示数据已准备（避免空页面）
- [ ] 剪辑工具已安装（剪映免费版即可）

## 视频→图文降级路径

当用户表示视频制作困难（配音、拍摄、剪辑）时，及时从视频方案切换到图文方案。**图文的生产效率和传播效果通常优于卡在半路上的视频。**

### 识别"视频障碍"信号

| 用户表达 | 翻译 |
|----------|------|
| "拍不好" | 怕录出来不专业 |
| "像流水账" | 不知道怎么组织画面 |
| "配音效果不好" | 声音表现力不够 |
| "剪辑太麻烦" | 时间成本高 |
| "先做图文吧" | 图文是更舒服的起点 |

### 降级策略

**不要浪费前期工作。** 已有的视频脚本可以直接拆解为图文笔记：

```
视频脚本（4-5min，6 个章节）
    │
    ├── 小红书笔记 1（功能 ①：Dashboard）→ 5 张截图 + 正文
    ├── 小红书笔记 2（功能 ②：Skill 系统）→ 3 张截图 + 正文
    ├── 小红书笔记 3（功能 ③：持久记忆）→ 2 张截图 + 正文
    ├── 小红书笔记 4（功能 ④：多平台接入）→ 3 张截图 + 正文
    └── 小红书笔记 5（功能 ⑤：发布频率）→ 2 张截图 + 正文
```

**每篇图文制作时间约 30 分钟**（截图 10min + 写文案 20min），5 篇连续发布可以撑起 1 周的更新节奏。

### 公众号长文

5 篇小红书发完后，可以整理为一篇**公众号长文**，作为"完整版"放在网站引流。

### 第 6 步：拆解小红书笔记

**每篇笔记结构**：

| 图片 | 内容 | 目的 |
|------|------|------|
| 图 1 | 封面：痛点 + 结果 | 吸引点击 |
| 图 2 | 场景概览/痛点展示 | 展示价值 |
| 图 3 | 案例拆解/代码展示 | 证明可行性 |
| 图 4 | 效果对比/工具推荐 | 增加收藏 |
| 图 5 | CTA：引导到网站/私信 | 引流 |

**标题公式**：
- `[结果]！[方法] 的 [数量] 个 [场景]，[人群] 也能上手`
- `别再 [痛点] 了！[解决方案]`
- `[身份] 教你：如何用 [方法] [结果]？`

**正文结构**：
```
痛点描述（2-3 句）
解决方案概述（1 句）

核心内容（3-5 点，每点简短）

工具清单（可选）

互动引导（评论/私信）
```

**标签建议**：
- 主标签：#AI 自动化 #效率提升
- 副标签：#程序员日常 #副业 #时间管理
- 长尾标签：#Python 编程 #职场效率 #自动化脚本

### 第 6 步：发布与互动

**发布时间**：
- 小红书：工作日晚上 8-10 点（流量高峰）
- 公众号：工作日中午 12-1 点 或 晚上 8-10 点

**互动策略**：
- 积极回复评论（前 1 小时最关键）
- 引导评论扣关键词（增加权重）
- 私信引导（获取完整资源）

### 第 7 步：数据复盘

**关键指标**：

| 指标 | 目标 | 说明 |
|------|------|------|
| 小红书互动率 | ≥5% | 点赞+收藏+评论/阅读 |
| 小红书收藏率 | ≥3% | 收藏是"有用"信号 |
| 网站引流点击率 | ≥2% | 笔记→网站转化 |
| 首周粉丝增长 | ≥50 | 小红书自然增长 |

**复盘问题**：
- 哪篇笔记互动率最高？为什么？
- 哪个标题/封面点击率高？
- 用户最感兴趣的内容是什么？
- 引流效果如何？

## 小红书封面设计实践

基于真实对标数据分析（3 个 AI 领域博主，12 条内容），得出以下封面设计规律：

### 暗色底 > 亮色底
所有高互动笔记（1000+ 赞）的封面均为暗色底（#0F172A 深蓝黑系）。**暗色封面在信息流点击率比亮色高 30-50%**。推荐配色：
- 背景：#0F172A 或 #000000
- 主文字：#FFFFFF（白色）或 #60A5FA（蓝色）
- 亮点字：#FBD209（金色）用于核心数字/关键词
- 标签底：#334155（半透明）

### 尺寸规范
- 小红书封面：**1080 × 1440 px**（3:4 竖屏）
- 标题字体：**80px+**（占封面 40% 以上面积）
- 导出时确保不小于 1080 宽，避免平台压缩模糊

### 设计工具链
| 工具 | 适用场景 | 时间 |
|------|---------|------|
| Canva（参数精确复制） | 需要科技风/精致感 | 5 分钟 |
| Excalidraw（手绘风格） | 需要差异化/亲切感 | 直接导出 |
| AI 绘画（通义万相/Midjourney） | 需要大片感 | 出图不稳定 |

### AI 绘画封面提示词模板
```
小红书封面，3:4 竖屏，纯黑底色，
中央大字 "[标题]" 白色，
紧接下方 "[亮点字]" 金色 (#FBD209)，
顶部小字 "[品牌标注]" 白色，
底部小字 "[副标题]" 灰色。
极简科技风，只有文字排版，高对比度，干净醒目。
```

## 观点类纯文字笔记（Opinion Piece 专属流程）

适用于 AI 焦虑/个人反思/行业观点类内容。和目标明确的教程类内容不同，观点类内容的核心是**情绪共鸣**而非信息传递。

### 何时使用纯文字格式

| 信号 | 说明 |
|------|------|
| 内容核心是感受/反思 | 配图反而稀释情绪浓度 |
| 结尾是开放式问题 | 不需要 CTA 按钮 |
| 对标账号为观点类（如卡兹克） | 不依赖封面驱动点击 |
| 用户明确说"不要封面" | 尊重用户的审美判断 |

### 7 段式观点结构

| 段 | 功能 | 字数 | 关键 |
|----|------|------|------|
| ① 钩子 | 制造认知冲突，一句话抓人 | ≤50 | 可以是反直觉观点（"学得慢就不用学了"） |
| ② 身份 | 建立共鸣，让读者知道"谁在说" | ≤50 | 越具体越好（"程序员 Jerry" > "一个普通人"） |
| ③ 焦虑/经历 | 用具体细节建立信任 | ≤150 | 荒诞细节 > 抽象描述（"养过龙虾" > "尝试过很多事"） |
| ④ 转折 | 从经历引出洞察 | ≤100 | 用隐喻/比喻（金矿、铲子）让抽象道理可视化 |
| ⑤ 核心观点 | 全文的"那句话" | ≤80 | 一句话能被截图传播 |
| ⑥ 现状/改变 | 给希望但不鸡汤 | ≤80 | 不需要完美答案，需要真实信号 |
| ⑦ 互动 | 引导评论 | ≤30 | 开放式问题，不预设答案 |

**总字数**：400-550 字

### 结尾策略：开放式 vs 行动信号

**开放式结尾**（推荐用于观点类）：
- 让读者停在问题里，而非答案里
- 例："谁知道呢？" / "你呢？" / "你有没有过这种感觉？"
- 效果：驱动评论，读者会在评论区替你把话说出来

**行动信号结尾**（适用于教程类）：
- 给具体可执行的下一步
- 例："打开 XX 试一下" / "从今天开始问自己这个问题"
- 效果：驱动收藏和行动

⚠️ **不要强行给开放式内容加行动信号**。如果稿子的情绪落点是"无力感"或"反思"，加了答案型结尾会破坏整篇的节奏。让读者和作者一起停在问题里——这恰恰是观点类内容最有力量的地方。

### 纯文字发布格式

小红书正文不支持空行，连续换行会被吞。用分隔符替代：

```
第一段内容...
.
第二段内容...
.
第三段内容...
```

每段之间用单独的 `.` 或 `—` 占一行，视觉上和空行效果一致。

### 已验证的高效结构（2026-05-12 更新）

| 模式 | 走向 | 适用场景 | 验证来源 |
|------|------|---------|---------|
| 情绪递进 | 钩子→身份→焦虑清单→发现→观点→状态→开放式结尾 | 观点/反思类（"AI让我焦虑"） | 第2篇实践 |
| 事实反转 | 普遍认知→个人查证→反直觉发现→分类→结论→互动 | 分析/质疑类 | — |
| 三层分类 | 钩子→身份→"我身边大概有三种人"→逐类分析→核心洞察→互动 | 观察/分析类（"AI取代程序员"） | 第3篇实践 |

### 纯文字为默认

对于观点类/个人反思内容，**纯文字是默认格式**，不是降级方案。封面图在观点类内容中会稀释情绪浓度——读者看到设计过的封面会预期"干货"，点进来发现是个人感受反而失望。纯文字让读者预期和内容匹配。只有当用户主动要求封面时才生成。

### 迭代写作流程（2026-05-12 精炼）

1. **用户提交原始草稿**（连续想法，未经分行；可能偏长或有结构问题，但情感真实、有自己的声音）
2. **助手做结构性诊断**：
   - 先列出**亮点**（必须保留的段落/句子/概念），让用户知道什么是对的
   - 再指出**结构问题**（松散/重复/缺收束/段落功能重叠），每次只聚焦 2-3 个
   - 给优化框架（7段，每段标注功能和预期字数）
3. **用户根据框架修改** —— 用户保留控制权，自己决定保留什么、改什么
4. **助手轻量润色**：只做分行、删重复、调节奏 —— **不动原话和原意**。AI 的角色是编辑，不是代笔
5. **用户确认终稿** —— 当用户说"可以"或"这是我的最终版"，停止给修改建议

⚠️ 迭代中注意：
- 不要一次给太多建议（用户会 overwhelmed）
- 每次只聚焦 2-3 个关键问题
- 当用户说"这是我的最终版"时，不要继续给修改建议——确认即可
- ⚠️ **用户自己发布**：不要主动提供"生成发布版文案"或复制粘贴服务，除非用户明确要求。用户更倾向于自己操作发布环节。

## 常见陷阱

### ❌ 过度优化
花太多时间设计图片，内容本身质量不高。

**✅ 正确做法**：内容 > 形式。先保证内容有价值，再优化形式。

### ❌ 一次性发布多篇
同一主题连续发布多篇，用户疲劳。

**✅ 正确做法**：间隔 1-2 天发布，保持节奏。

### ❌ 忽略互动
发完就不管，不回复评论。

**✅ 正确做法**：前 1 小时积极互动，增加权重。

### ❌ 引流太硬
每篇都硬推网站，用户反感。

**✅ 正确做法**：2-3 篇内容中穿插 1 篇引流，自然引导。

### ❌ 强迫做对比/虚构数据
当用户只深度使用过工具 A 没用过工具 B 时，强行做"A vs B"对比会显得虚假。

**✅ 正确做法**：真诚告知用户"我只深度用过 A，B 我没做够评估"。改为聚焦 A 的优点展示，不做对比。真诚反而加分。

### ❌ 强迫视频
用户表达视频制作困难（配音不自然、怕像流水账、拍摄难度大）时，不要坚持做视频。

**✅ 正确做法**：识别"视频障碍"信号 → 主动建议转为图文笔记。已有的视频脚本可以拆解为多篇图文，不浪费前期工作。

### ❌ 给开放式内容强行加 CTA
观点类/反思类内容的情绪落点是"问题"而非"答案"。强行在结尾加行动信号（"试试XX"）会破坏整篇节奏。

**✅ 正确做法**：如果稿子以"谁知道呢？""你呢？"收尾，不要画蛇添足。开放式结尾驱动评论，封闭式结尾驱动收藏——选哪个取决于内容类型。
