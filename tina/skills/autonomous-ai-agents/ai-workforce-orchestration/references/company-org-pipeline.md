## 事件驱动流水线 (5 步闭环)

```
Step 1: 管理层讨论
  └── Demon (挑战) + Alex (方向) + Karen (战略对齐) + Allen (可执行性) + Jack (时间线)
      └── 产出: 讨论结论 + 范围定义

Step 2: Allen 拆解 + 分配
  └── 把讨论结论拆成可执行任务 (30-60min 粒度)
  └── 每个任务附带验收标准 + 指定执行人
      └── 产出: 任务清单 (谁做什么、验收条件、预计工时)

Step 3: 执行 + Checker 迭代 (最多 3 轮)
  └── 执行人 (Justin/Eli/Rose) 产出
  └── 对位 Checker 独立检查 → 打回或通过
  └── 通过后进入下一步
      └── 产出: 验证通过的交付物

Step 4: 管理层汇总
  └── Jack (进度) + Alex (是否符合目标) + Allen (任务完成率)
  └── 汇总成最终报告
      └── 产出: 验收报告 (状态灯+进展+风险+行动项)

Step 5: Jerry 验收
  └── 你点头 → 交付
  └── 你打回 → 回到 Step 2 或 Step 3
```

## 角色能力映射方法论

当需要决定"谁该做什么"时，用这个流程:

1. **列出候选角色** — 从管理层角色库查
2. **读对应 SOUL** — 主要看第一条思维法则（核心能力）
3. **提取关键词** — eg. "规格拆解→任务"、"优先级"、"ROI"、"方向"
4. **对位问题** — 当前需求是什么？是拆任务？定方向？管时间线？
5. **排除不匹配** — SOUL 里没有相关描述的，明确排除（记下来为什么）
6. **得出分配结论** — 最优 + 备选

方法论示例 (本 session):
> 问题: "谁负责分任务?"
> 候选: Alex / Allen / Jack / Karen / Demon
> 读 SOUL → Allen 第1条"把规格说明书拆成可执行任务" ✅
> 排除: Alex (PM管方向不管分配), Karen (战略层), Demon (不碰执行), Jack (管周期太粗)
> 结论: Allen 主理, Jack 辅助

## Checker-per-Profile 设计原则

- 每个执行角色有且只有一个对位 Checker
- Checker **不改**产出，只验证
- Checker 打回时附带具体理由（不是"不过"而是"第X部分格式不对"）
- 同一任务最多迭代 3 轮，仍不过 → 上报 Jerry
- Checker 的模型可以和执行人相同（已验证不同 prompt 比不同模型效果更显著）
