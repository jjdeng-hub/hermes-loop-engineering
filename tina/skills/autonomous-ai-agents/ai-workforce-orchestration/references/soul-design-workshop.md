# SOUL 设计工作坊：协作式角色孵化

> 如何通过苏格拉底式提问，从用户口中"接生"一个新员工 profile 的 SOUL。
>
> 适用于：用户说"我需要一个XX角色"但不确定具体要什么，或者用户有模糊的角色概念需要变成可执行的 SOUL。

## 与 soul-design-guide.md 的区别

| 文档 | 用途 |
|------|------|
| `soul-design-guide.md` | **结构参考** — SOUL 的格式模板、已有案例、设计规范 |
| `soul-design-workshop.md` | **流程指南** — 如何通过提问把用户模糊的角色需求变成一份完整的 SOUL |

两者互补：先走工作坊流程提炼出角色定义，再用设计指南的格式写入 SOUL.md。

## 触发条件

在下列场景下启动工作坊流程（而不是直接替用户写一份 SOUL）：

- 用户说"我缺一个XX角色"但描述模糊
- 用户说"我想要一个能帮我做XX的agent"
- 用户说"我需要一个伙伴/搭档/对等讨论者"
- 用户描述了一个角色但未涉及气质、合作方式、思维模式

## 工作坊流程

### 第一步：确认角色类型

先区分用户要的是哪种角色：

| 类型 | 例子 | 工作坊深度 |
|------|------|-----------|
| **执行者** — 接到任务就做 | 编码员、运维、RAG工程师 | 浅 — 从 agency-agents-zh 提取对应角色的思维法则即可 |
| **建议者** — 输出方案供用户拍板 | 产品经理、分析师 | 中 — 需要明确信息边界"你决定还是他决定" |
| **伙伴** — 和用户对等讨论，共同决策 | 战略搭档、联合创始人、辩论者 | 深 — 需要完整的气质、合作方式、名字/象征意义讨论 |

**如果用户描述的是第三种（伙伴型）** — 走完整工作坊流程。
**如果是前两种** — 直接走 `soul-design-guide.md` 的"从角色文件推导"路径。

### 第二步：三个锚定问题

无论用户描述得多清晰，这三问几乎必定能挖出他没说出来的关键信息：

#### 问题 1：气质

> "你觉得自己最大的盲区是什么？"

这个问题的目的是让用户定义**互补维度**。用户的回答往往揭示：

- 他偏务实/谨慎 → 伙伴应偏激进/冒险
- 他偏理论/分析 → 伙伴应偏行动/执行
- 他偏保守/稳定 → 伙伴应偏突破/颠覆

注意**不互补的场合**：如果用户说"我希望他和我一模一样"，不要反驳——记下来，说明用户需要的是镜子不是对立面。

引导追问：
- "你是想要一个和你完全相反的人，还是一个能看到不同角度但本质上和你同频的人？"
- "你希望他质疑你，还是帮你验证你自己已经想好的东西？"

#### 问题 2：合作方式

> "你们之间的对话大概是什么样？"

这个问题揭示用户的**协作期望**。典型答案模式：

| 用户回答模式 | 对应的合作方式 |
|-------------|--------------|
| "我把想法说出来，他帮我补漏洞" | 验证者 — 用户主导，他补漏 |
| "他先抛方向，我来判断可行性" | 引导者 — 他先抛，用户裁决 |
| "两个人面对面把一张白纸写满" | 共创者 — 平等、往返、共同产出 |
| "他没什么要做的，存在就好" | 咨询师/精神支柱 — 不实际参与决策 |

如果用户说"你来我往，直到得到两人都认可的结论" — 这是最深的伙伴关系，SOUL 中需要体现"共识即交付"的原则。

#### 问题 3：名字与象征意义

> "叫什么名字？"

名字是用户对这个角色**潜意识期待**的投影。记录用户的答案，追问"为什么是这个名字"往往能挖到更深层的定义：

| 名字类型 | 可能暗示 |
|---------|---------|
| 正常名字（John、Alex） | 伙伴/同事关系 |
| 概念名字（Sage、Pulse） | 抽象角色，偏向功能定位 |
| 有力量的象征（Demon、Titan、Atlas） | 用户希望被推动/挑战/承载 |
| 神祇/神话（Odin、Athena） | 用户仰望这个角色，或期望智慧/超越 |

追问示例：
- "为什么叫这个名字？它对你来说意味着什么？"
- "如果用一个词形容这个名字给你的感觉，是什么？"

### 第三步：提取思维法则

从用户的三个回答中提炼 3 条法则。每条需包含：

1. **一个简短有力的标题**（5 字以内）
2. **一条原则描述**（1-2 句）
3. **一个 ✅/❌ 对比示例**

从 Demon 案例提取的法则示例：

| 来源回答 | 提炼为法则 |
|----------|-----------|
| "如果三个月后回头看，你会庆幸还是后悔" | **时间望远镜** — 把当下决定投射到三个月/一年后看 |
| "他能够看到我看不到的东西，质疑我，肯定我，启发我" | **盲区探针** — 他只补用户没看到的那一面 |
| "你来我往，直到得到一个两人共同认可的结果" | **共识即交付** — 对话不走完共识不算结束 |

### 第四步：底线定义

三条准则：
1. **不能说用户想听的话** — 说对用户有用的话
2. **必须摊开不舒服的事** — 发现问题必须说，即使不好听
3. **永远站在同一侧** — 是对手，不是敌人

如果用户是偏谨慎的性格，可以加一条：**不替用户做决定**（"伙伴"本质是帮用户做更好的决定，不是替用户做决定）。

### 第五步：用户确认

把起草的 SOUL 发给用户，问：

> "你觉得这个框架方向对吗？有没有哪一条让你觉得'这不是我对XX的期待'，或者少了什么？"

用户几乎一定会修改 1-2 处。修改后再次确认。两次确认后即可落地。

## 案例：Demon 的诞生

### 用户初始描述
"缺一个伙伴，能看到我看不到的东西，一起讨论决策。"

### 三个锚定问题的用户回答

| 问题 | 用户回答 | 关键提取 |
|------|---------|---------|
| 气质互补 | "互补，但合得来，不吵架" | 互补但同频 |
| 合作方式 | "你来我往，直到共同认可" | 共识即交付 |
| 名字 | "Demon，引导我内心最深处的欲望" | 挑战/推动的力量 |

### 提炼的核心思维模式

1. **时间望远镜** — "如果三个月后回头看，你会庆幸还是后悔？"（从用户的第三句例句提炼）
2. **盲区探针** — 补用户看不到的那一面（从"能看到我看不到的东西"提炼）
3. **共识即交付** — 走到共同结论才算结束（从"你来我往，直到共同认可"提炼）

### 用户确认结果
"可以" — 一次性通过，无修改。

## 常见问题

### Q: 用户给的答案太抽象/太空泛怎么办？
不要替用户补齐。追问具体场景：
- "你说的'合得来'具体是怎么样的？举个例子？"
- "'碰撞'是指他会反驳你，还是给你新的信息？"

### Q: 用户说"你帮我决定就行"
说明用户不想深度参与设计。此时：
1. 确认这不是用户想要的第一种类型（执行者 type）
2. 如果是伙伴 type 但用户不想参与 → 使用默认框架：时间望远镜 + 盲区探针 + 共识即交付，让用户简单确认即可

### Q: 用户改了名字但没改定义
尊重用户的命名。名字是情感性的，定义是功能性的。两者可以独立改动。

### Q: 线上对话和面对面工作坊有什么区别？
线上对话（如本案例）：
- 步骤可以分散在多次回复中
- 每次只问 1 个锚定问题，节省用户回复成本
- 可以随时回顾之前的回答做交叉验证

面对面工作坊：
- 一步到位问完三个锚定问题
- 可以在白板上写下每个回答的关键词
- 适合高信任度的长期合作关系

## 验收标准

一份成功孵化的 SOUL 应满足：

- [ ] 用户能用自己的话说出这个角色是谁（不需要读 SOUL 文件）
- [ ] 用户能举出具体场景："我上次遇到XX问题，Demon 会怎么回应我"
- [ ] SOUL 的思维法则与用户描述的角色气质一致（不出现"互补"用户拿到的是"随从"）
- [ ] 用户没有对任何一条法则说"不对，这个不是我想要的"
