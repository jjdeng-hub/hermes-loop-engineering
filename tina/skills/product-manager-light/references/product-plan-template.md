# 产品计划模板（PM Checker 验证通过的参考格式）

> 参考 2026-06-20 知识库产品 V2 计划（8 条件 checker 以 PASS 条件通过）。
> 每次 PM 输出计划时，用此结构 + 8 条件 checklist 自动验证。

## 文档结构

```markdown
# [产品/功能名] — 执行方案

## 1. 问题定义
- 解决谁的问题？
- 当前怎么被解决的？（痛点）
- 一个具体场景描述

## 2. 用户画像
- **Persona A：[名称]** — 特征、痛点、购买力
- **Persona B：[名称]** — 特征、痛点、购买力
- （最多 3 个，第 3 个是"延迟满足"型用户，不做当下决策依据）

## 3. 核心方案
- 做什么，不做什么（明确边界）
- 技术选型及理由
- 为什么选这个方案（vs 其它选项）

## 4. 定价策略（若涉及）
- 方案 A（推荐）：价格 + 理由
- 方案 B：价格 + 适用场景
- 不做的定价方案及理由
- MVP 阶段的支付/交付方式

## 5. 执行计划
- 总耗时估计（小时/天）
- Phase 1（1-2周）：核心功能
- Phase 2（2-4周）：上线 + 验证
- Phase 3（4周+）：迭代
- 每阶段：做什么、产出什么、谁验收

## 6. 风险矩阵
| 风险 | 概率 | 影响 | 缓解方案 |
|------|:----:|:----:|---------|
| [风险描述] | 高/中/低 | 高/中/低 | [具体方案] |

## 7. 成功标准（3 个以内）
1. [可衡量的成功指标]
2. [可衡量的成功指标]
3. [可衡量的成功指标]
```

## Checker 验证要点（PM 特定）

V2 计划通过 checker 的要点：

1. **用户画像必须具体** — 不要泛泛的"目标用户"，要有具体场景和痛点。Persona A 负责"立即付费"，Persona B 负责"未来流量"。
2. **必须对比方案** — 展示考虑过哪些选项以及为什么选当前方案。Checker 会查"假设验证"条件。
3. **风险矩阵要有缓解方案** — 只列风险没有缓解方案会被判 FAIL。V1 就死在这里。
4. **MVP 支付方式必须可行** — 不要假设"上支付系统"。MVP 用人工方案（如微信转账）是可接受的。
5. **定价必须覆盖成本** — "永久买断"型定价需要算 operating cost。如果算不过来，就不做这个定价档位。

## 参考：V2 计划概要

实际通过的 V2 计划（知识库产品）：
- **MVP**: RAG 查询接口（非 PDF 打包），用户发问题 → 系统从知识库检索答案
- **定价**: A/B 测试 9.9元/单次查询 vs 29.9元/24h不限量
- **不做**: 99元永久（operating cost 覆盖不了）
- **支付**: MVP 走微信手动转账
- **时长**: 60h 总工时（约 1-2 个月业余时间）
