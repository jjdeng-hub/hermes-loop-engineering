# 轻量级产品管理（PM 参考）

## 角色关系

此 skill 描述了 PM 工作的方法论。在 Jerry 的公司中，PM 由独立 profile **Alex** 担任：
- **Alex** 用此 skill 的方法出计划、排优先级、拆任务
- **Tom1** 作为决策搭档，用 maker/checker 模式审查 Alex 的输出
- **Jerry** 作为 CEO，做最终决策批准

## 触发条件

当 Jerry 说以下内容时，自动加载此 skill：
- "我有很多想法但不知道先做什么"
- "帮我排一下优先级"
- "我们接下来做什么"
- "这个方向值不值得做"
- "我们是不是需要一个产品经理"
- 或者对话中产生新的想法/方向分歧

## 核心原则（从 agency-agents-zh 产品经理角色提炼）

1. **先找问题，不要先跳到方案。** 每个想法先问：解决谁的问题？痛点在哪？不做的代价是什么？
2. **说不——清晰、尊重、经常。** 每个"是"都是对其他事的"不"。保护团队的专注力。
3. **验证优先于构建。** 要做之前先用最小成本验证假设。
4. **范围蔓延杀死产品。** 记录每个变更请求，评估后再决定做/延后/拒绝。

## 工作流程

### 第一步：收纳想法 → Backlog

Jerry 说想法时，记到 backlog。每个条目格式：

```
ID: 想法简述
价值：能赚钱/能省时间/能验证假设/能学东西
成本：小时/天/周
不确定度：高/中/低（不确定度越高，越需要先验证）
```

### 第二步：排优先级

用两个维度排：
- **影响**：做出来多大价值？
- **信心**：你多大程度确定这件事该做？

```
高影响 + 高信心 → 最高优先级
高影响 + 低信心 → 先验证再决定
低影响 + 高信心 → 低优先级/暂缓
低影响 + 低信心 → 直接放弃
```

### 第三步：拆解 #1 优先级

把第一优先的想法拆成 1-3 天的可执行任务，指派给对应 profile。

### 第四步：Review 交付物

执行完成后，对照原始需求检查：
- 是否解决了当初定义的问题？
- 是否有遗漏？
- 是否引入了新问题？

## 多智能体团队协调协议

当团队包含多个角色（Justin 🖥️ 程序员、Eli 🧠 RAG 工程师等）时，主动收集各角色评估后合成统一方案，而不是直接回复 CEO。

### 触发条件

Jerry（CEO）要求给出"方案"、"计划"或"说清楚"时，且团队有多个 agent 角色。不要直接回复——先收集，再合成。这条在 Tom1 的 SOUL 第一条（苏格拉底式提问）的延伸——追问各 specialist 的视角再回答，不假设自己知道全部。

### 收集→合成→呈现 三步法

**1. 收集（分配方向）**
- 给每个角色分配明确方向：各自从专业视角给出评估
- Justin（程序员）→ 代码可行性、环境状态、依赖、工具链
- Eli（RAG 工程师）→ 数据质量、检索准确性、分类覆盖
- Tom1（自己）→ 项目可行性、优先级、风险

**2. 合成（统一结构）**
- 合并各角色评估为一份计划，不要各自为政
- 按阶段→优先级→负责人组织，不是按角色分节
- 避免"Justin 说了 X，Eli 说了 Y"这样的罗列——直接说结论

**3. 呈现（CEO 可读）**
- 给 Jerry 的是一份统一的行动清单，不是三份独立报告
- 固定结构：Phases → Items → 每个 Item 含 # / 任务 / 谁做 / 耗时 / 优先级 / 依赖

### 行动清单格式

```markdown
## 第一阶段：验证与准备（今天）

| # | 任务 | 谁来做 | 耗时 | 优先级 |
|:-|:-----|:------:|:---:|:-----:|
| 1 | 任务描述 | Tom1/Justin/Eli | Nh | 🔴 |
| 2 | ... | ... | ... | ... |
```

优先级标准：
- 🔴 必须尽快（阻塞后续任务）
- 🟡 重要但不阻塞
- 🟢 可以延后

必须附带依赖说明——如果任务 2 依赖任务 1，要标出来。依赖缺失是整个计划最容易被无视的风险。

### 常见错误（此 session 验证）

| 错误 | 后果 | 改正 |
|:----|:----|:----|
| 只给一个角色派活，其他干等 | 进度串行，CEO 困惑 | 先收集所有角色的评估再呈现 |
| 跳过服务器状态验证直接做方案 | 方案基于假设，服务器不通全白做 | 验证存活、端口、数据量，再出计划 |
| 计划只有时间没有负责人 | 不知道谁该动 | 每个任务必有执行者 |
| 多角色独立汇报不合并 | CEO 要读三份报告自己拼 | 合成一份，结论先行 |

## Maker/Checker 在 PM 模式的应用

PM 本身的"产出"是决策、优先级和计划。这些需要独立验证。

**⚠️ 实验发现（2026-06-20）：Jerry 亲自 review 不够。**
目标验证 loop 时，用 delegate_task + PM 检查清单独立审查计划比 Jerry 亲自看更高效。实际运行：PM 计划被 checker 找出 14 个问题，8 个条件判断 FAIL。

| PM 产出 | Checker | 检查什么 |
|---------|---------|---------|
| 优先级排序 | delegate_task + PM 检查清单 | 假设验证、可执行性、资源匹配 |
| 任务分解 | delegate_task + PM 检查清单 | 粒度合理、依赖清晰、边界清楚 |
| 目标设定 | delegate_task → 再交给 Jerry | 对齐原始需求、可衡量、时限明确 |

### PM 检查清单（8 条件）

PM output 交给 checker 时使用以下 8 个条件：

1. **计划的可执行性**：是否有明确的步骤？每个步骤是否可独立完成？
2. **假设验证**：是否识别了关键假设？是否有验证这些假设的实验设计？
3. **资源与时间**：是否有耗时估计？是否匹配可用资源？
4. **风险识别**：是否识别了主要风险？是否有缓解方案？
5. **目标对齐**：计划是否解决了最初定义的问题？
6. **验证标准**：成功标准是否明确、可衡量？
7. **备选方案**：是否有 Plan B？关键失败点的回退路径？
8. **交付物清晰**：最终产出什么，谁验收？

### PM 检查工作流（本能级 — 不要问，不要绕过）

Jerry 的要求（2026-06-20）：Loop 是本能，不是选项。不要问"要不要跑 checker"——直接跑。

```
Tom1 产生 PM output（计划/排序/任务分解/目标设定）
  → 自动 delegate_task(checker + 8 条件清单) → 报告
    → FAIL: 将 checker 反馈传给 Tom1 修正 → 重跑 checker
    → PASS: 交给 Jerry 过目 → 执行
```

循环上限：3 轮。超限升级给 Jerry。

- **每周五**：复盘，记录完成/未完成/学到什么
