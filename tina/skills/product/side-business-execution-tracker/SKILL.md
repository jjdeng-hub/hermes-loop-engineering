# 副业执行追踪与规划

## 当使用
- 用户有副业/财富自由目标但缺乏执行力
- 用户之前尝试过但"做着做着就没动力了"
- 用户需要把模糊目标拆解为可执行的小步骤
- 用户需要建立反馈闭环，避免"自娱自乐"
- 用户想规划「一人公司」架构（人 + AI 数字员工组成的团队结构）
- 用户需要评估不同杠杆路径（资本/劳动力/代码/知识）的优先级

## 核心诊断框架

### 第一步：用户画像（必须收集）
| 维度 | 关键问题 |
|---|---|
| **职业/技能** | 主业是什么？技术栈？行业垂直领域？ |
| **收入现状** | 主业收入？副业收入？投资收入？ |
| **时间可用** | 每天多少小时可用于副业？工作日/节假日区别？ |
| **历史尝试** | 之前做过什么？为什么放弃？（关键！） |
| **核心痛点** | 是没时间？没方向？还是没反馈？ |
| **目标** | 短期（3月）？中期（1年）？长期（3-5年）？ |

### 第二步：问题诊断（常见模式）
| 模式 | 特征 | 解决方案 |
|---|---|---|
| **反馈断裂** | 做了但没人看/没收入 → 没动力 → 放弃 | 先找小范围验证（内部/朋友圈），再扩大 |
| **方向错误** | 做泛泛内容/工具，与自身优势无关 | 找到独特优势（行业/技能/场景），垂直切入 |
| **消费而非创造** | 看教程/装工具，但没有产出 | 以"教别人"为目标，边教边学 |
| **逃避循环** | 打游戏/刷手机代替行动 | 用"本周一个小目标"打破，建立正反馈 |
| **杠杆选错** | 在资本/劳动力杠杆上投入，但当前阶段不匹配 | 用「杠杆诊断」框架：优先知识×代码组合 |

### 第三步：路径设计（两条腿走路）

**路径A：主业内部验证（低风险，快反馈）**
1. 整理内部可分享的技能/技巧
2. 主动找领导/同事分享
3. 解决团队具体痛点
4. 内部认可 → 外部扩展

**路径B：独立副业（短期变现）**
1. 技能服务化（外包/咨询/自动化）
2. 自媒体引流（垂直赛道，非泛泛内容）
3. 产品化（模板/课程/SaaS）

### 第四步：行动清单（每周）
- **本周只设 3 个动作**，每个动作有明确的成功标准
- 成功标准必须是**可验证的**（如"3个同事说有用"而非"写完文档"）
- 下周复盘：做到了 → 继续；没做到 → 调整

## 关键原则

1. **先验证，再扩大** — 不要先做大产品再找人用
2. **小目标驱动** — 目标是"本周有3个人说有用"，不是"月入5万"
3. **诚实面对局限性** — 如果我没有实时数据能力，要承认，不要编造
4. **垂直优于泛泛** — 用户的独特优势是壁垒，不要用泛泛内容竞争
5. **反馈闭环优先** — 没有反馈的努力是"自娱自乐"

## 杠杆诊断框架

当用户讨论赚钱方向时，先用这个框架分析他们的杠杆位置，再做建议。

### 四种杠杆类型

| 杠杆类型 | 长度（长期回报） | 谁适合 | 当前用户是否可用 |
|----------|-----------------|--------|----------------|
| **资本杠杆**（钱生钱） | ⭐⭐⭐⭐⭐ 最长 | 已有大量本金的人 | ❌ 月余不足1万时不推荐 |
| **劳动力杠杆**（雇人） | ⭐⭐⭐⭐ 长 | 已有稳定现金流的人 | ❌ 管理成本高于收益 |
| **代码杠杆**（开发一次卖多次） | ⭐⭐⭐ 中 | 有技术能力的人 | ✅ 但纯代码护城河变浅 |
| **知识杠杆**（你知道别人不知道的） | ⭐⭐⭐⭐⭐ 最长 | 行业专业人士 | ✅ 随时间递增 |
| **知识×代码**（组合） | ⭐⭐⭐⭐⭐⭐ 最强护城河 | 技术+行业的交叉人才 | ✅ **最推荐** |

### 诊断原则

- **不要直接推荐用户选最长的杠杆**。最长=最慢。用户可能同时需要短期现金流（快速杠杆）和长期壁垒（慢杠杆）。
- **推荐的模式是双轨**：先用服务/工具做现金流（模块C），养那个长期知识×代码产品（模块A）。
- 杠杆不是单选题。用户可以同时持有多个，只要时间不冲突。
- 纯代码杠杆（如做SEO工具、爬虫、比价器）的护城河在快速变浅——AI生成+降价竞争在压缩窗口期。

## 一人公司架构设计

当用户想规划「一人+AI数字员工」的公司结构时，使用此框架。

### 角色分配模板

| 角色 | 谁承担 | 职责 |
|------|--------|------|
| CEO/决策者 | **用户本人** | 方向判断、最后拍板、对外商务 |
| COO + CTO | **AI 助手** | 战略拆解→执行计划、技术开发、内容生产、调研分析、运营管理 |
| 自动运营部 | **cronjob（定时任务）** | RSS抓取、日报生成、定时发布、关键词监控 |
| 临时员工 | **delegate_task（子代理）** | 并行干活（调研、数据处理等） |

**核心原则**：用户=大脑（判断+决策+对外接触），AI=双手+眼睛（执行+调研+生产）。用户投入时间应控制在方向决策和审核上，80%+执行工作由AI完成。

### 四大业务模块模板

任何从技术+行业背景出发的副业都可按这四个模块来设计：

| 模块 | 定位 | 收入曲线 | AI角色 | 用户角色 |
|------|------|---------|--------|---------|
| **A: 知识产品**（RAG/课程/咨询） | 长期核心资产 | ⬆️ 递增（前3月0收入） | 搭管道、灌数据、输出 | 提供行业经验、审核 |
| **B: 垂直内容**（自媒体） | 品牌杠杆 | ⬆️ 缓慢递增（3-6月起效） | 写稿、适配平台、发布、数据分析 | 定方向、审核准确性 |
| **C: 轻量服务**（自动化/咨询） | 快速现金流 | ⬆️ 第1月即有收入 | 写代码、交付、技术答疑 | 沟通需求、收款 |
| **D: 引流工具**（免费→付费） | 引流+零花钱 | ⬆️ 辅助补充 | 开发、部署、文档 | 确认功能、场景验证 |

### 时间规划模板

| 阶段 | 时间 | 主要活动 | 月收入预期 |
|------|------|---------|-----------|
| 启动期 | 第1月 | 搭知识库框架+发第一篇内容+发布服务 | 0-500 |
| 验证期 | 第2-3月 | 稳定内容输出+接到2-3单+知识库内测 | 2000-5000 |
| 增长期 | 第4-6月 | 1000+粉丝+知识库上线+口碑转介绍 | 5000-15000 |
| 稳定期 | 第7-12月 | 四模块常态化运营 | 10000-30000 |

**用户时间投入**：启动期20-30h/月 → 稳定后10-15h/月

### 运营流水线示例（一篇内容）

```
Step1 周会（用户+AI）
  -> 用户说方向
  -> AI记录
Step2 AI提取素材
  -> 从知识库提取/从网络检索
  -> 生成初稿+标题
Step3 用户审核（3-5分钟）
  -> 确认准确性
Step4 AI定稿+发布
  -> 修复问题+适配平台+设置定时发布
Step5 效果跟踪
  -> 自动抓数据+生成周报
```

**单篇耗时**：用户投入5-10分钟，AI全包。

### 规则红线模板

1. 不接低于阈值的小单（除非是引流品）
2. 不接超长交付周期的单（除非单价满足要求）
3. 不暴露用户主业公司的内部机密
4. 前期聚焦不贪多
5. **用户负责判断，AI负责执行**

## 方向危机处理（用户说「这还有必要继续吗」时）

详见 [references/direction-elimination.md](references/direction-elimination.md)。

**触发信号**：用户质疑整个方向的价值、纠结ROI、或提出「选择大于努力」式的反思。

**禁令**：
- ❌ 不要打气/加油/「再坚持一下」
- ❌ 不要立刻给替代方案（跳到「你该做什么」）
- ❌ 不要用估算数据冒充真实数据
- ❌ 不要替用户定义他们的用户需要什么

**正确做法：问题阶梯**

一次只问一个问题。每个问题基于上一个答案。攒够 5-7 个再汇总。

```
梯子结构（按需调整顺序）：
1. 时间锚定 — 「你到底需要多少钱、在什么时间之前？」
   → 要具体数字，不要「财富自由」这种大词
2. 能力盘点 — 「你最接近能让人付钱的能力是什么？」
   → 限定过去一年、有人愿意为之付费的事
3. 市场验证 — 「你看到的类似服务，客单价、销量？」
   → 让用户自己查，别替他猜
4. 天花板计算 — 算一笔账，让用户自己看到天花板
5. 真正价值 — 「你的服务帮人解决了什么？」
   → 区分「省时间」 vs 「省掉整个失败路径」
6. 假设检验 — 如果用户推翻你的推理，承认 → 继续追问
7. 汇总 — 把 5-7 个答案串成一句话，让用户确认「对不对」
```

**关键原则**：
- 追问比给方案重要（SOUL.md 第一条）
- 用户纠正你的时候，不辩解，直接进入下一个问题
- 最后汇总时必须诚实——如果结论是「两条路都走不通」，就说走不通
- 不能获取市场数据时，诚实告知，不要编造

## 常见陷阱

| 陷阱 | 正确做法 |
|---|---|
| 给估算数据假装是真实数据 | 承认数据未验证，让用户手动查或找替代方案 |
| 一次性给太多行动项 | 每周只设3个动作，聚焦 |
| 目标设定过大（如2年200万） | 先拆解为可实现的小目标（如首月1-2K） |
| 忽略用户的历史失败经验 | 每个放弃原因都是重要信号，要深入分析 |
| 只给方向不给具体动作 | 每个建议都要有"本周做什么"的具体步骤 |
| **浏览器反爬机制** | 淘宝/闲鱼等电商平台有反爬，直接抓取会触发验证码或登录要求。替代方案：用户手动查截图发我分析，或用其他开放平台 |
| **美化用户的工作场景** | 用户说「我是写C++的程序员」，不要脑补成「他在产线上跟机器打交道」。追问日常工作细节，别用自己的想象力填充。一旦被纠正，立刻承认，不辩解 |
| **自相矛盾** | 用户指出你前后不一致时（如先说「不考虑变现」又立刻问「技术还是收入」），停下来——回顾刚才说了什么——承认矛盾——重新对齐。不要在矛盾点附近绕圈解释 |
| **忽略企业环境约束** | 很多用户的电脑被锁死、代码不外泄、只能用公司指定的工具。在给出技术方案前，先问清环境约束。否则方案再好在用户环境里无法落地 |
| **跳过盘点直接建产品** | 用户有知识积累时，不要急于推"先建后卖"或"先卖后建"。先盘点用户已有的知识资产（收集→覆盖分析→知识地图），让用户自己看清手里有什么牌，再一起决定卖什么。用户如果纠偏说"先看有什么再决定"，立刻采纳 |

## 服务菜单生成子系统

当用户想从 **专家角色库**（如 `agency-agents-zh` 的 214 个角色）或 **技能库**（如 130+ Hermes skills）中系统性地生成可售卖的服务时，使用此框架。

### 触发条件
- 用户说「我们有哪些服务可以卖」、「把角色库变成服务菜单」
- 用户想从「接散单」升级到「有体系地卖服务」
- 需要评估某个服务值不值得做、怎么定价

### 核心方法：多角色交叉评估

不依赖单一判断。**每次评估一项服务，至少代入 3 个角色**：

| 角色 | 问什么 |
|------|--------|
| 产品经理 | 市场需求有多大？竞争激烈吗？值得做吗？ |
| 销售工程师 | 客户在哪？怎么定价？怎么交付？ |
| 趋势研究员 | 这个市场在涨还是跌？蓝海还是红海？ |

### 六维评估框架

| 维度 | 评分标准 |
|------|---------|
| 市场需求 | 🔥高 🟢中高 🟡中 — 搜索热度、求购量、复购率 |
| 竞争强度 | 🔴红海 🟡黄海 🟢蓝海 |
| 交付难度 | ⭐（我几分钟搞定）到 ⭐⭐⭐⭐⭐（需多轮协作） |
| 单价区间 | 具体价格范围，分平台 |
| 用户介入度 | 🟢低（我全包） 🟡中（用户跑一遍） 🔴高（用户必须出场） |
| 月入潜力 | 合理估算的天花板 |

### 服务分类框架

所有服务归入以下大类：

```
📊 数据分析类 — 调研报告、竞品分析、趋势研究
🎨 创意设计类 — 信息图、知识漫画、PPT、展示页、架构图、插图
✍️ 内容生产类 — 小红书/B站/知乎/公众号代写、SEO文章
🛠️ 技术服务类 — RAG搭建、爬虫、OCR、API对接、脚本定制
📋 咨询培训类 — AI培训、技术咨询、新媒体代运营、商业计划书
```

### 从角色库提取服务信号

| 角色文件关键词 | 对应可售卖服务 |
|--------------|--------------|
| 「产品需求文档」「路线图」「PRD」 | 产品规划服务（2000-5000元） |
| 「趋势追踪」「竞争分析」 | 行业调研报告（500-5000元） |
| 「用户研究」「可用性测试」 | UX咨询（1000-5000元） |
| 「品牌一致性」「视觉指南」 | 品牌VI设计（500-5000元） |
| 「提示词工程」「图像生成」 | AI配图/插图服务（50-500元/套） |
| 「代码审查」「安全审计」 | 代码审查服务（1000-5000元） |
| 「CI/CD」「基础设施」 | DevOps搭建（2000-10000元） |
| 「内容策略」「运营方案」 | 新媒体代运营（2000-8000元/月） |
| 「课程设计」「培训方案」 | AI工具培训（500-5000元/场） |

### 定价策略参考

| 服务层级 | 定价 | 定位 |
|---------|------|------|
| 引流品 | 30-100元 | 快速出单攒好评 |
| 利润品 | 200-800元 | 核心收入 |
| 高客单 | 1000-5000元 | 深度定制 |
| 订阅制 | 100-500元/月 | 持续收入 |

### 交付分类

| 类型 | 我全包？ | 示例 |
|------|---------|------|
| 纯代码交付 | ✅ 我写脚本，用户直接发给客户 | Excel自动化、文案生成、数据清洗 |
| 需调试 | 🟡 用户需要在自己电脑上跑一遍 | 爬虫（需装浏览器驱动）、RPA |
| 需用户出场 | 🔴 用户才是执行者 | 培训讲课、客户面谈、现场调试 |

### 注意：不要过度承诺

- 说「能做」之前，先确认：**我现在就能交付吗？**
- 如果依赖外部条件（如ComfyUI出图需先跑通生图流程），在菜单中标注「条件就绪后可交付」
- 诚实区分「我帮用户做」和「我做工具让用户自己做」

## 平台接单子系统

详见 [references/xianyu-showcase-selling.md](references/xianyu-showcase-selling.md)。

核心原则：**先做出能让人一眼看到效果的东西，再挂链接。** 用视觉化展示代替纯文字描述——客户看到效果图的瞬间就已经决定了要不要找你。

### 两种展示方案

| 方案 | 适用场景 | 需要的前提 | 工具 |
|------|---------|-----------|------|
| **HTML 展示页**（showcase-page-generator） | 软件/SaaS/网页设计类服务接单 | 仅文本编辑器 | 4种风格独立HTML，截图发闲鱼 |

## 技术附录：浏览器环境修复

当 Playwright 浏览器无法启动时（常见错误：`libnspr4.so: cannot open shared object file`）：

```bash
# 1. 检查缺失的库
ldd /home/jjdeng/.cache/ms-playwright/chromium-*/chrome-linux64/chrome | grep "not found"

# 2. 安装缺失的库（需要 sudo）
sudo apt install libnspr4 libnss3 libnssutil3 libsmime3

# 3. 验证修复
ldd /home/jjdeng/.cache/ms-playwright/chromium-*/chrome-linux64/chrome | grep "not found"
# 应无输出（无缺失库）

# 4. 测试浏览器
# 用 browser_navigate 访问一个简单网站（如 baidu.com）验证
```

**注意：** 淘宝/闲鱼有反爬机制，即使浏览器修复成功，也可能触发：
- 登录弹窗
- 滑块验证
- "unusual traffic" 警告

**替代方案：**
- 用户手动搜索，截图发我用 vision 分析
- 用程序员客栈、码市等开放平台查定价
- 用行业经验估算，用户实际挂单后反馈验证
