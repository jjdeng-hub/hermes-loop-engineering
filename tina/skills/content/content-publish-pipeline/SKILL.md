# Content Publish Pipeline

Jerry 抖音内容创作到发布的完整管线。每条新内容按阶段推进，不跳过、不遗漏。

## 触发条件

用户提出新选题或新内容开始创作时自动激活。

---

## 管线阶段

每阶段完成后向用户确认，再推进下一阶段。

### Phase 0 — 启动检查

- **先跑 `date`** 确认当前日期时间（不可跳过！多次日期混乱的根因）
- 读 `.cheat-state.json` 获取当前队列和发布节奏
- 确认下一篇编号、发布日期（隔天发，19:00-19:30 全年不变）
- 注意：用户工作日 8:00-21:00 不在家，此期间仅 QQ 可用，无法操作电脑
- 发布必须在 19:00-19:30，不要擅自改为其他时间

### Phase 1 — 选题与写作（含排版，不可拆分）

- 发散→收敛：列出所有候选，最后统一筛选
- 选题来源：AIHOT 热点 + 用户真实体感 + 选题库
- 用户确认选题后，由对话中的思想碰撞自然产出内容
- **新分工模型（#15 后确立）**：用户主导写作和修改，我负责对照规则扫描底线。不替他写初稿，不替他拟标题，不替他加过渡。我只做镜子和底线检查。
- **初稿前先做重合检测**
- **规则先行**：列规则清单，用户确认再动笔
- **⚠️ 定稿 = 输出排版版**：粗体/斜体/引用/分隔线全部到位后才说「定稿」。不允许先定稿后补排版——这是 #14 第四次犯这个错

### Phase 1.1 — 原文检查（条件触发）

出初稿前，如果选题依赖外部新闻：
- 先查 AI HOT 是否有 `summary`
- 如果 `summary` 为 null，把原文链接给用户，等用户看过确认后再写
- **不要基于二手转述写初稿**——用户说过「先把这篇新闻的完整内容给我看一下」
- AI HOT API 调用方式：纯 `curl` + `--max-time 10`，避免管道到 Python（WSL 环境下 `curl | python3` 可能被工具层拦截）

### Phase 2 — 盲测与预测

- 定稿后进行，不要跳过
- 确定 Bucket 预测：退步 / 持平 / 命中 / 小爆 / 大爆 + 中枢值
- 至少 1 个锚点对照 + 核心赌注 + 风险
- 写入 prediction 文件到 `predictions/` 目录

### Phase 3 — 封面（与 Phase 1 同步，不后补）

#13 验证：两张 CEO 脸 + 标题的封面方案，估算多带来 1000-2000 浏览。封面决定了读者点不点——你在抖音上只有一张缩略图的机会。

**流程**：Phase 1 定稿 → 同时出封面方案（不晚于保存文件）→ 用户确认 → 制作

**方案可以是想法**：不要求每次都生成成品图。定稿时至少说出封面的核心思路——用图还是纯文字、什么视觉元素、为什么跟这篇文章匹配。

两种方式，用户选择：

**A / AI 生成封面（当前在用）**
- 用户用智谱 CogView 等工具生成封面图，主题与文章内容匹配
- 图上需包含标题文字（清晰可读，适配抖音缩略图）
- 风格可以每篇不同，匹配文章情绪
- 通常不需要额外叠加文字，如需署名用 PIL 叠加 `@Jerry在想什么`
- **风格选择**：写实电影感 vs 漫画夸张风——根据文章调性判断。观点尖锐、意象大胆（如大猩猩拿枪）→ 漫画夸张风更匹配，降低暴力感、增加传播性。写实沉重主题 → 电影感
- **漫画夸张风 prompt 要素**：夸张比例（枪管比手臂粗）、反差萌（危险物品+无害表情）、小细节破严肃（枪口绑蝴蝶结）、暖色底+强对比、上方三分之一留白

**B / 桌面实拍 + 文字叠印（备用）**
- 脚本：`generate_cover.py`（blur=15, 92%宽, 微软雅黑Bold）
- **多行标题**：超过 8 字自动换两行，用 `\\n` 分隔
- **关键词引号**：用「」包裹关键词制造视觉焦点
- **背景轮换**：`backgrounds/bg_*.jpg` 随机选（需 ≥5 张）
- 纯主标题，不加标签和副标题
- 署名 `@Jerry在想什么` 右下角

**选择原则：** 8 个粉丝阶段封面一致性不重要，读者认的是观点和文风。可混合使用，通过数据判断哪种效果好。

### Phase 5 — 话题

- 抖音 ≤5 个
- 2 精准圈层 + 2 情绪/话题 + 1 泛流量

### Phase 6 — 保存与登记

- 发布文件：`publish/YYYY-MM-DD_NN_slug.md`（含排版版正文 + 元数据）
- **唯一保存路径**：`/mnt/c/Users/jjdeng/Desktop/xiaohongshu-content/publish/`
- ⚠️ 不得在其他目录（~/、/tmp/、项目目录等）创建内容文件。此路径是所有内容文件的唯一来源，查文件先来这里
- 目录结构自检：保存后确认文件出现在 `ls /mnt/c/Users/jjdeng/Desktop/xiaohongshu-content/publish/` 中
- 更新 `.cheat-state.json`（如存在）
- 更新 USER.md 发布队列

### Phase 7 — 复盘（T+2，不可跳过）

- 发布后第 2 天自动提醒
- 对比预测 vs 实际：浏览 / 赞 / 评 / 藏 / 享
- 偏差分析 → 更新 lessons → 如有必要 bump rubric
- 写入 retro_notes 到 cheat-state + prediction 文件

### Phase 8 — 口播练习（进行中）

- 每日 20:00 自动提醒（cron job `797207909733`）
- 当前阶段：Step 1 录音念稿
- 技巧：手机放桌上，嘴离手机 20cm，稿子举在手机后面 — 减少呼吸声
- 同一篇稿子念 3 遍 > 3 篇各念 1 遍
- 已发布内容优先练（内容已验证，复用风险最低）
- 后续阶段：录屏+配音 → 桌面+手+配音 → 出镜（不露脸）

### Phase 1.5 — 证据升级链合并（条件触发）

当定稿后当天出现直接验证该篇核心论点的大新闻时，合并而非分开发新篇：

- **判断标准**：外部新闻直接验证个人故事，且体量/权威明显大于个人案例
- **结构**：个人 → 验证 1（同类）→ 验证 2（跨量级巨头）→ 全球信号
- **桥接句式**：「不止我们。」→「更不止我们和XX。」→「体量差一百倍，结论一模一样。」
- **效果**：从「某个人的抱怨」升级为「全行业都在算这笔账」
- 参考：content-writing-patterns §9 证据升级链

## Rubric v2

- 站队型：Composite ×1.2，赞 ×2.0，藏 ×2.0
- 提问型：赞 ×0.3，评 ×3.0
- 安抚型：藏 ×0.3
- **反问 ≠ 提问**：答案读者点进来就知道 → 站队型
- 标题闭环结构（结尾=标题）→ 增强立场感

## 用户规则

- 发布时间：19:00-19:30 全年不变
- 发布节奏：当前日更实验中（5/28 起），原本隔天发 2-3 篇/周。日更前提：质量不降。如互动率下滑 → 退回隔天
- 粉丝基数小（<10）时日更不会互相抢流量，多发=多曝光
- 图文消费规律：大部分浏览在第二天白天（上班摸鱼场景，图文隐蔽可刷，视频不行），约 1/3 在当晚（晚上用户更倾向视频）。19:00 发布 = 当晚吃一波 + 次日白天主力推流，是图文内容的最优窗口
- 工作日 8:00-21:00 不在家，此期间仅 QQ 可用，无法操作电脑
- 每次新对话先跑 `date`，确认当天日期和星期
- 查数据先读 `.cheat-state.json`，不要搜会话记录
- **标题格式**：主标题 10 字左右，副标题 20 字左右，陈述句式。不用问句标题。用户提出的标题即为最终版——不迭代替换。教训：#15 被叫「起名废」，根源是用户自己说了标题后仍替他迭代
- **⚠️ 大推流下一篇挤压**：#8→#9、#13→#14，降幅均 87%。上一篇持续推流时新篇初始曝光被挤占。如上一篇仍在推流，下一篇隔 2 天

## 写作规则

- **避免过度引用前文**：最多开头一处衔接（如「周二刚写完……」），后面每段独立成立，当读者第一次读你。不要反复用「我之前说过」「前几篇里写过」
- **模型名不挂版本号**：只说 GPT、Gemini、Claude，不写 GPT-4o、Gemini 2.5。版本号变化太快且用户可能看到不同版本
- **专业严谨，不凭空断言**：有数据引用数据，没数据说逻辑推理，不编造因果关系（如「开源降价闭源一定跟」→「历史上反复发生：云计算、数据库、操作系统……」）
- **受众共鸣优先**：Opus 国内封号需代理，多数用户无体感。优先用 GPT、Gemini 等国内可用的模型举例
- **不编造用户情况**：我不确定的事直接问，不要把假设写进文章
- **结尾必须有力量**：不用安慰型收尾（「你还有时间」「别急」）。用画面感或压迫感收——具体意象（「那条线，每年都在往下移」）、对比（「企业等的不是技术突破。是价格跌破你的工资线」）、或把球踢回给读者。结尾是读者最后带走的东西，不能用「只是……」软着陆
- **例子节奏**：多例结构中 2 个是甜点——足够建立模式，不产生重复感。开头例子 2-3 句就够（说清楚+点出矛盾），把篇幅留给后半段的个人视角和金句收束。不要每个例子都展开成独立段落。4 个例子的结构会产生疲劳，必砍
- **过渡句逻辑自洽**：检查递进关系是否成立。「从A到B到C」要求 C 不在 B 的集合内。发现逻辑裂缝直接改表述（如「从个案到行业共识」）
- **不插表情/符号**：严肃观点型文章不加 emoji 或颜文字，会泄掉叙述的压迫感。轻松吐槽类可以考虑
- **观点先行 > 例子先行**：正文第一句就扔判断（如「有人跟你说AI替代人类，有人说AI只是工具。两个说法都太简单了。」），不要从行业案例铺垫入题。读者先被观点砸中，再往下看你怎么证明
- **开头不突兀**：如果从具体行业切入（如「医疗这边」），前面必须有一句过渡（如「先看两个离普通人最近的行业」），否则像从半截开始。但优先用观点先行，避免这个问题
- **例子上限 2 个**：多例结构中，2 个行业/案例是节奏甜点——足够建立模式，不产生重复感。4 个必砍。砍完后篇幅还给后半段的个人视角和收束
- **标签自检**：定稿后逐标签核对——每个标签必须对应文中实际出现的行业或人群。砍了内容就必须砍对应标签。"AI"和"人工智能"不能同时用（同义重复），2-2-1 的两个精准标签必须覆盖不同圈层
- **小红书 10 标签**：抖音用 5 个（2-2-1），小红书可用满 10 个
- **反焦虑内容传播力弱**：#10 盲测翻车验证——缓解焦虑的内容（「AI太贵替代不了你」）触达远低于制造焦虑（「AI要有身体了」）。算法偏好恐惧 > 安慰。选题时评估情绪的传播方向，不是内容质量，是情绪方向决定触达
- **ER 维度高估风险**：反焦虑内容容易被我高估 ER——我把内容共情当传播动力，但算法不看这个。以后反焦虑选题自动降 ER 1 分
- **发布前扫 AI 味**：小红书读者比抖音更敏感 AI 味，同一篇内容抖音无人指出、小红书第一个评论就是批评。发小红书前额外做一次 AI 味检查
- **小红书发前额外做 AI 味检查**：小红书读者对 AI 味比抖音敏感得多。同一篇内容在抖音无人指出，小红书第一条评论就是劈头盖脸的批评。同步到小红书前必须比抖音版多扫一遍——重点查「注意…」「不是…而是…」句式和三段论论证体
- **窄选题+无个人锚点=天花板**：#10（1802）和 #14（550）验证。纯行业新闻转述、无个人体感锚点 → 算法判定「仅特定群体感兴趣」→ 初始推流一小波 → 数据平 → 打住。选题评估加一条：给一个不用AI写代码的人看前三句，他看不完就是你窄

## 选题防重合

Phase 1 出初稿前增加一步：
- 如果新选题的核心论点 70%+ 与上一篇已发内容重合，问用户：调整角度还是换选题？
- 不是禁止写同一赛道——但如果论证过程大量重复，读者感受差
- 信号：新稿的核心金句跟前一篇高度相似 → 重合风险高
- **案例**：#11「DeepSeek降价75%」vs #10「AI太贵」。两者都讲"AI成本→企业算账→替代人"，核心论证路径相同。用户选择 shelve #11 草稿到 scripts/，另开新选题 → #12「人类与AI的分界线」社会观察型，完全不同的切入角度
- shelve 的草稿不删除，等未来角度翻新或隔足够远（≥3篇）再考虑

## 内容类型

当前已验证的类型：
- **个人经历+观点**（#1-#10）：以公司/个人真实经历为锚点，展开行业观点。用户有独特资格说
- **社会观察型**（#12）：不依托个人经历，纯观察各行各业。结构：观点当头 → 2 个具体行业切入 → 个人视角落地（周报/会议）→ 金句收束。选题面广，需案例支撑

## 常见遗漏

| 遗漏项 | 后果 | 预防 |
|--------|------|------|
| 盲测未做 | 复盘无对照，rubric 无法校准 | Phase 2 不可跳过 |
| 姿态分类错误 | 预测偏差巨大（如 #8：提问型 → 实际应站队型，112 赞远超 5-12 预测） | 反问→站队型规则检查 |
| 复盘遗忘 | 数据积压，校准停滞 | T+2 自动触发 |
| 日期混乱 | 发布节奏错乱，「昨天」「明天」指代错误 | Phase 0 先跑 `date`，不凭记忆推断 |
| 发布时间被改 | 用户发布铁律 19:00-19:30 被擅自改成其他时间 | 记住：全年不变。⚠️ 用户随口说「晚上X点发」通常是「我X点有空操作」的意思，不表示改发布时间。听到此类表述时**先反问确认**，不要直接改元数据 |
| 时间措辞混乱 | 在傍晚时说「今晚只剩」，但用户刚确认已经是晚上 | 用户说「现在刚七点」→ 当前就是晚上。不要用「今晚」指代未来时段，用「接下来」或确认具体时间 |
| **排版遗漏** | 定稿发用户时忘了加粗/引用/斜体，用户提醒后才补 | Phase 3 排版环节定稿后直接产出排版版，不分两次 |
| **规则未先列导致多版返工** | 写完才对照规则，每版都发现违规要改 | Phase 1 确认选题后立即列规则清单，用户确认再动笔 |
| **封面后补** | 发布前临时拼凑，错过点击窗口 | Phase 1 定稿同时出封面方案，不晚于保存文件 |
| **标题过度迭代** | 用户说了标题后被反复替换，激怒用户（#15「起名废」） | 用 `content-creation-distribution` 技能的观点/热点类标题公式 [事件]+[态度] 产出 3-5 个，他选。不替他决定，不迭代替换。验证：#16「终于有个大佬把话说清楚了」用公式一次选中。不靠直觉设计，靠公式驱动 |
| **窄选题无锚点** | 推流天花板极低（#14: 550, #10: 1802）。Copilot定价对非程序员无意义 | 选题评估：给不用AI写代码的人看前三句，看不完就是窄 |
| **经验铁律化** | #15 总结的 10 条经验被当成铁律套到 #16，用户纠正：「很多太绝对了，上篇合适的下一篇不一定合适」 | 每篇新稿子独立判断。经验是参考，不是规则。写作规则（禁止清单、噪音、中文节奏）是硬约束；迭代经验（过渡、收束、意象选择）是软参考 |
| **预测后改标题不同步** | 改了标题但只改了文章，prediction 文件和 state 文件残留旧标题——#16 实战发现 | 标题变更时同步更新三处：① `publish/` 文章文件 ② `predictions/` 预测文件 ③ `.cheat-state.json`。三者必须一致 |
